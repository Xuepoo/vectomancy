# Curve Smoothing Experiment Design

## Objective

Reduce the "staircase/aliasing" effect in the generated equations from raster images. The current pipeline uses Zhang-Suen thinning and RDP simplification, which leaves discrete integer coordinates connected by straight lines.

## Approach

Implement and compare two distinct smoothing algorithms on separate Git branches to evaluate their visual fidelity and mathematical complexity.

### Branch 1: `feature/smoothing-chaikin` (Chaikin's Corner Cutting)

- **Concept**: Smooth the sharp corners of the RDP-simplified polyline by iteratively cutting off corners.
- **Implementation**: Add `chaikin_smooth(points: &[Point2D], iterations: usize) -> Vec<Point2D>` in `vectomancy/src/math/mod.rs`.
- **Integration**: Apply this function right after `simplify_rdp` in the math or parser pipeline (e.g., `vectomancy/src/parser/raster.rs`).
- **Pros**: Extremely fast, purely geometric, retains the existing `LineTo` spline builder logic.

### Branch 2: `feature/smoothing-bezier` (Catmull-Rom to Cubic Bezier)

- **Concept**: Convert the discrete points into mathematically smooth Cubic Bezier curves.
- **Implementation**: Modify `vectomancy/src/math/spline.rs` to support `CatmullRom` spline generation. Calculate control points for smooth interpolation between consecutive points, translating them into `BezierSegment::CubicTo`.
- **Integration**: Replace the naive `LineTo` sequence generator with this new spline fitter.
- **Pros**: Produces mathematically true curves, reducing the total number of distinct equation segments and providing infinite smoothness at render time.

## Error Handling & Edge Cases

To ensure robustness, both approaches must implement the following error handling:

- **Insufficient Points**: If RDP simplification yields fewer than 3 points for a path, the smoothing algorithms (both Chaikin and Catmull-Rom) must safely return the original points without panicking or modifying the topology.
- **Collinear Points/Zero Length Segments**: Algorithms must detect and filter out or safely handle zero-length segments to avoid division by zero during tangent/control point calculations.
- **Invalid Inputs**: Any malformed `Point2D` arrays passed to the math module must return a clear `VectomancyError` rather than causing an unexpected crash.

## CLI Integration & Final Selection

The generated algorithms will be evaluated. Based on the evaluation, if both are deemed valuable (e.g., Chaikin for speed, Bezier for high quality), we will introduce a CLI flag `--smooth-mode <MODE>` taking values `chaikin`, `bezier`, or `none`. For the experiment, they will simply replace the default logic in their respective branches.

## Concrete QA Scenarios & Evaluation Metrics

Both branches will be tested against `vectomancy/tests/assets/miku.png` (and other assets). The generated Python scripts will be output to `/tmp/agents/vectomancy/` and evaluated against the following concrete metrics:

1. **Visual Fidelity**: Does the output remove the staircase aliasing without blurring or destroying the original topological structure of the image? (Manual visual inspection of generated plots).
2. **Topological Correctness**: Do closed loops remain closed? Do corners intended to be sharp get overly smoothed out?
3. **Script Size & Equation Count**: Compare the final size of the generated `.py` script and the number of math equations generated by each approach.
4. **Rendering Speed**: Measure the time taken by the Python backend (e.g., matplotlib) to render the equations. Does the `CubicTo` approach render faster than a dense array of `LineTo` points from Chaikin?
